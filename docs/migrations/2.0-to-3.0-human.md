# Upgrading from cached 2.x to 3.0

cached 3.0 is a major release. The largest changes are a new disk backend (redb replaces sled),
a reworked redis feature layout that splits the runtime and TLS choices onto their own axes, and a
public-API cleanup across the store constructors, traits, and macro attributes.

This is the human walkthrough of the high-traffic changes. For the complete, exhaustive list of
every breaking change with detection greps, see the
[agent-oriented guide](2.0-to-3.0.md).

## Before you upgrade: toolchain requirement

cached 3.0 requires Rust >= 1.89 (up from 1.85), because redb 4.x needs it. Update your toolchain
if you are below 1.89. Any Rust >= 1.89 works; you do not need the latest stable.

## Quick summary

Macro attribute changes (most common):
- `size = N` is removed. Use `max_size = N` (the 2.x deprecation alias is gone).
- `ttl = N` (bare integer seconds) is removed. Use `ttl_secs = N`, `ttl_millis = N`, or a
  `Duration` expression string `ttl = "Duration::from_secs(N)"`.
- `unbound` is removed. A bare `#[cached]` already builds an `UnboundCache`.
- `refresh = true` now requires a TTL (previously it was silently ignored without one).

Feature renames (check your `Cargo.toml`):
- `disk_store` -> `redb_store`.
- Redis runtime and TLS are now separate feature axes (details below).
- `redis_ahash`, `wasm`, and `async_tokio_rt_multi_thread` are removed.

Store and type renames:
- `DiskCache*` -> `RedbCache*` (no alias kept).
- `ShardedCache*` -> `ShardedUnboundCache*` (no alias kept).

Construction and trait imports:
- I/O store `builder()` methods take the primary required field positionally (`RedbCache::builder(name)`, `RedisCache::builder(prefix)`, `AsyncRedisCache::builder(prefix)`).
- The short method aliases (`get`/`set`/`len`/...) moved to extension traits; `use cached::prelude::*;`
  brings everything in.
- `CachedAsync` is renamed to `CachedGetOrSetAsync`.

Behavior:
- The default `#[cached]` write behavior is unchanged from 2.x (no synchronization).
  `sync_writes = "by_key"` is a new opt-in.
- Serialize-backed stores self-heal undecodable entries by default (opt out with
  `strict_deserialization(true)`).

## Step-by-step upgrade

### 1. Macro attribute changes

`size` becomes `max_size`:
```rust
// Before
#[cached(size = 100)]
// After
#[cached(max_size = 100)]
```

`ttl = <integer>` becomes one of three forms:
```rust
// Before
#[cached(ttl = 60)]
// After (whole seconds, preferred)
#[cached(ttl_secs = 60)]
// After (milliseconds, new)
#[cached(ttl_millis = 500)]
// After (arbitrary Duration expression, must be a string literal)
#[cached(ttl = "cached::time::Duration::from_secs(60)")]
```
The three forms are mutually exclusive. The same forms apply to `#[once]` and
`#[concurrent_cached]`. Builder `.ttl(Duration)` is unchanged, and builders gain `.ttl_secs(n)` /
`.ttl_millis(n)` shorthands.

`unbound` is gone (it was redundant), and `refresh = true` now needs a TTL:
```rust
// Before (refresh silently did nothing without a ttl)
#[cached(refresh = true)]
// After
#[cached(ttl_secs = 60, refresh = true)]
```

### 2. Feature renames and the redis axis split

Rename `disk_store` to `redb_store`:
```toml
# Before
cached = { version = "3", features = ["disk_store"] }
# After
cached = { version = "3", features = ["redb_store"] }
```

Redis is now split into three independent axes: a runtime, an optional TLS backend, and optional
capabilities. Pick one from each axis you need:
- Runtime: `redis_tokio` or `redis_smol` (required for any async redis use).
- TLS (optional): `redis_tokio_native_tls` / `redis_tokio_rustls` (or the `redis_smol_*` twins).
  The runtime features no longer imply native-tls, so a `rediss://` URL needs an explicit TLS
  feature.
- Capabilities (optional, additive): `redis_async_cache` (client-side caching) and
  `redis_connection_manager` (auto-reconnect). Neither pulls a runtime on its own, and
  `redis_connection_manager` is now a per-cache opt-in via `.connection_manager(true)` rather than
  a global swap.

```toml
# Before (2.x): the capability implied the tokio runtime
cached = { version = "2", features = ["redis_connection_manager"] }
# After: pair the capability with a runtime, opt in per cache
cached = { version = "3", features = ["redis_connection_manager", "redis_tokio"] }
```

`redis_ahash`, `wasm`, and `async_tokio_rt_multi_thread` are removed. The first gated no cached
code; `wasm` was unnecessary (wasm-compatible time is transparent); and for multi-threaded async
tests, add `tokio = { version = "1", features = ["rt-multi-thread", "macros"] }` to your own
`[dev-dependencies]`.

### 3. Store and type renames

The sled-backed disk store is now redb-backed and named `RedbCache`:
```rust
// Before
use cached::stores::DiskCache;
// After
use cached::stores::RedbCache;
```
The old `DiskCache` / `DiskCacheBuilder` / `DiskCacheError` / `DiskCacheBuildError` names are
removed (no alias). Existing on-disk caches are not read across the format change (entries
recompute on first access); the old sled directories linger as dead files and can be deleted
manually.

`ShardedCache` is renamed to `ShardedUnboundCache`:
```rust
// Before
let c: ShardedCache<String, u32> = ShardedCache::new();
// After
let c: ShardedUnboundCache<String, u32> = ShardedUnboundCache::new();
```

### 4. Store construction

The I/O-store builders now take the primary required field positionally:
```rust
// Before
let b = RedbCache::<String, String>::new("my-cache");
let b = RedisCache::<String, String>::new("prefix", ttl);
// After
let b = RedbCache::<String, String>::builder("my-cache").build()?;
let b = RedisCache::<String, String>::builder("prefix").ttl(ttl).build()?;
```
The redis TTL is now optional; omit `.ttl(...)` to store keys without expiry.
A missing required field now returns `Err(BuildError::MissingRequired(...))` from `build()` rather
than failing at construction.

Two builder renames on the in-memory stores:
- `.refresh(...)` -> `.refresh_on_hit(...)`.
- `.capacity(n)` (the preallocation hint) -> `.initial_capacity(n)`, distinct from `.max_size(n)`
  (the eviction bound).

New `::new()` conveniences exist on the in-memory and sharded stores when you want a default
configuration:
```rust
let cache: LruCache<String, u32> = LruCache::new(100);
let cache: TtlCache<String, u32> = TtlCache::new(Duration::from_secs(60));
```

### 5. Trait imports

The short method aliases (`get`, `set`, `remove`, `len`, `is_empty`, ...) moved off the `Cached` /
`ConcurrentCached` core traits onto blanket extension traits `CachedExt` / `ConcurrentCachedExt`.
The core traits keep only the `cache_`-prefixed methods. The simplest fix is the prelude:
```rust
use cached::prelude::*; // core + extension traits + the common concurrent traits
```
If you import traits individually, add `CachedExt` (or `ConcurrentCachedExt`) alongside the core
trait, or call the `cache_`-prefixed method directly.

Two related splits, both aimed at letting you import the concurrent traits together without
`E0034` "multiple applicable items in scope":
- The shared concurrent helpers (`cache_size`/`len`/`is_empty`, and the TTL controls) moved to
  `ConcurrentCacheBase` and `ConcurrentCacheTtl`. The prelude brings them in.
- `ConcurrentCachedAsync`'s cache methods gained an `async_` prefix (`async_cache_get`,
  `async_cache_set`, ...), so they no longer collide with the synchronous `ConcurrentCached`
  methods on a store that implements both.

The in-memory async-closure trait `CachedAsync` is renamed to `CachedGetOrSetAsync`:
```rust
// Before
use cached::CachedAsync;
let v = cache.async_get_or_set_with(key, || async { value }).await;
// After
use cached::CachedGetOrSetAsync;
let v = cache.async_cache_get_or_set_with(key, || async { value }).await;
```
Its four sync passthroughs (`async_cache_get`/`_set`/`_remove`/`_clear` on an in-memory store) are
removed; call the synchronous `Cached` methods directly (no `.await`).

### 6. `get_or_set_with` returns `&V`

`cache_get_or_set_with` / `cache_try_get_or_set_with` (and the async twins) now return `&V` /
`Result<&V, E>` instead of `&mut V`. Read-only call sites usually need no change. To mutate the
returned value, switch to the new `*_mut` variants (`cache_get_or_set_with_mut`, etc.).

### 7. `sync_writes` default is unchanged; `by_key` is a new opt-in

A bare `#[cached]` still runs concurrent first-calls for the same key independently, matching 2.x
(and Python's `functools.lru_cache`). If you want per-key deduplication so the body runs at most
once per key per miss window, opt in:
```rust
#[cached(sync_writes = "by_key")]
fn compute(x: u64) -> u64 { expensive(x) }
```
Note that the `"by_key"` bucket lock is held across the whole function body, so do not use it on
recursive or re-entrant memoized functions (colliding keys deadlock), and concurrent readers of
the same key serialize.

### 8. Error handling on serialize-backed stores

The store error enums (`RedbCacheError`, `RedisCacheError`, and their build-error twins) dropped
the redundant `Error` suffix from variant names, became `#[non_exhaustive]` struct variants, and
now carry boxed sources (`Box<dyn std::error::Error + Send + Sync>`) rather than concrete backend
types. An exhaustive `match` needs a wildcard arm.

By default, a serialize-backed store self-heals an entry it cannot decode: `cache_get` deletes the
offending entry and returns `Ok(None)` (a transparent miss the caller repopulates). If you would
rather have decode failures surfaced, opt into strict mode:
```rust
let cache = RedbCache::<String, Data>::builder("my-cache")
    .strict_deserialization(true)
    .build()?;
```
Redis values are now stored as MessagePack rather than JSON. Old 2.x JSON entries are read
transparently (a backward-read fallback) and rewritten as MessagePack on their next write, so a
rolling upgrade does not invalidate a shared redis cache.

## Additional breaking changes from the final 3.0 API

These changes landed after the rc.1 snapshot above. For full Detection/Action entries see the
[agent-oriented guide](2.0-to-3.0.md) entries #62-#66.

### `cache_set_ref` / `async_cache_set_ref` return `Result<(), _>`

The borrowed-set methods on `SerializeCached` / `SerializeCachedAsync` no longer return the
previous value. The return type is now `Result<(), Self::Error>`. This removes the per-write GET
on IO stores (Redis, redb).

If you have a custom `impl SerializeCached`:
```rust
// Before
fn cache_set_ref(&self, k: &K, v: &V) -> Result<Option<V>, Self::Error> {
    // ... write ...
    Ok(Some(prev))
}
// After
fn cache_set_ref(&self, k: &K, v: &V) -> Result<(), Self::Error> {
    // ... write ...
    Ok(())
}
```
To get the previous value, call `cache_get` before the set.

### `ConcurrentCacheBase::len` removed; `is_empty` -> `cache_is_empty`

`ConcurrentCacheBase::len` (an alias for `cache_size` returning `Result<Option<usize>, _>`) is
removed. Use `cache_size` directly. `ConcurrentCacheBase::is_empty` is renamed to `cache_is_empty`.

The concrete sharded types still have inherent `len()` / `is_empty()` returning plain `usize` /
`bool`; those are unaffected.

### Redis and redb `::builder()` takes a required positional argument

The primary identifier moves back to a positional argument on the cache type's `builder` method:

```rust
// Before (rc.1 form - builder().field(x))
let b = RedbCache::<String, String>::builder().name("my-cache");
let b = RedisCache::<String, String>::builder().prefix("my_prefix");
// After
let b = RedbCache::<String, String>::builder("my-cache");
let b = RedisCache::<String, String>::builder("my_prefix");
```

`RedisCacheBuilder::new()` and `RedbCacheBuilder::new()` (builder-struct constructors) still
take no arguments and chain with setters. Only the `CacheType::builder(x)` shorthand changed.

### Redis TTL is now optional

`RedisCache::builder("prefix").build()` no longer returns `MissingRequired("ttl")`. An unset TTL
means entries are written without an expiry command and never expire. Set `.ttl(Duration)` if you
want expiry.

### `#[concurrent_cached(disk = true, cache_prefix_block = ...)]` is a compile error

This combination previously compiled but silently ignored `cache_prefix_block`. It is now a
compile error. Remove `cache_prefix_block` from disk-backed annotations; disk stores use `name`
as the redb table identifier.

### `Resp2DowngradeWithClientSideCaching` requires `redis_async_cache`

The `RedisCacheBuildError::Resp2DowngradeWithClientSideCaching` variant is declared only when
the `redis_async_cache` feature is enabled (it was never constructible without it). If you
match on it, gate the arm with `#[cfg(feature = "redis_async_cache")]` or rely on the `_` arm
the `#[non_exhaustive]` enum already requires.

### `Return<T>` fields are now private

The `value` and `was_cached` fields on `Return<T>` (from `cached_proc_macro_types`) went private.
Code that accessed them directly produces `field value of struct Return is private` or
`field was_cached of struct Return is private` compile errors.

```rust
// Before (2.x): direct field access
let r: Return<String> = my_cached_fn();
println!("{}", r.value);
if r.was_cached { ... }
```

```rust
// After (3.x): deref, into_inner(), and was_cached()
let r: Return<String> = my_cached_fn();
println!("{}", *r);             // deref coerces to &T
println!("{}", r.into_inner()); // consumes r, returns T
if r.was_cached() { ... }
```

### `cache_contains` / `async_cache_contains` are now required on the concurrent traits

If you have a custom `impl ConcurrentCached` or `impl ConcurrentCachedAsync` for your own store,
you must add these methods. The `V: Clone` bound that was on the defaulted rc versions is gone.

```rust
fn cache_contains(&self, k: &K) -> Result<bool, Self::Error> {
    self.cache_get(k).map(|v| v.is_some())
}

async fn async_cache_contains(&self, k: &K) -> Result<bool, Self::Error> {
    self.async_cache_get(k).await.map(|v| v.is_some())
}
```

Note that the trait declares `async_cache_contains` in RPIT form
(`impl Future<Output = ...> + Send` with `where Self: Sized + Sync, K: Sync`); the
`async fn` above desugars to it, so either spelling works in an impl. The machine guide
shows the full RPIT form.

If you only use the built-in sharded, redis, or redb stores, no change is needed.

### `TtlSortedCache::set_ttl` inherent method removed

Call `set_ttl` through the `CacheTtl` trait (same as `TtlCache` and `LruTtlCache`):

```rust
use cached::CacheTtl; // or `use cached::prelude::*;`
cache.set_ttl(Duration::from_secs(30));
```

### `contains` on single-owner stores is now peek-based (rc-only behavior change)

`CachedExt::contains` delegates to `Cached::cache_contains`, which the built-in stores implement
with a side-effect-free peek. Previously it called `cache_get`, which counted a hit/miss, promoted
LRU recency, and refreshed TTL on refresh-on-hit stores. The new implementation does none of those
things and reports expired entries as absent. No code change is required; this is a behavioral
note for callers who relied on the side effects of the old `contains`.
