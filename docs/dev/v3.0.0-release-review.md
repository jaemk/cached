# 3.0.0 pre-release review: findings and plan

Review of everything accumulated since 2.0.2 (`817301b..HEAD`, currently 3.0.0-rc.5).
Three tracks: a sharded code review of the full release diff (6 reviewers), an external
consumer-experience review (throwaway crate in `local/`, compiler-verified), and a
cross-type parity audit of the public surface.

Verdict: the public API is consistent and ready. No breaking changes are required beyond
the decisions below. One high-severity behavior bug (B1) and the migration-guide staleness
(D1-D3 in "Docs") must land before the final release.

## Behavior fixes (must land before 3.0.0)

- B1 (high) `src/stores/lru_ttl.rs:346-351`: `set_entry` fires `on_evict` with the
  caller's key instead of the stored key. `LruCache::cache_set` returns `Option<V>` so the
  stored key is unrecoverable at the callback site. Fix: add a non-trait inherent method on
  `LruCache` that returns the displaced `(K, V)` and use it from `set_entry` (see decision
  DEC-2).
- B2 (minor) `src/stores/sharded/expiring.rs:476`, `src/stores/sharded/ttl.rs:652`:
  `cache_set` evaluates `is_expired()` on the displaced entry after releasing the shard
  write lock (TOCTOU at the expiry boundary). Evaluate while holding the lock.
- B3 (medium) `src/stores/redb.rs:1133,1172`: `remove_expired_entries_impl` takes two
  `SystemTime::now()` snapshots (scan pass and write pass). Use a single snapshot.
- B4 (minor) `src/stores/sharded/expiring.rs:149-153`,
  `src/stores/sharded/expiring_lru.rs:145-153`: `deep_clone` loads hit/miss counters after
  dropping the shard read guard, so the metrics snapshot can be newer than the entry
  snapshot. Move the loads under the lock (the other three sharded stores already do).
- B5 (low) `src/stores/redis.rs:1883`: `url_pins_resp2` compares `protocol=` values
  case-sensitively. Verify against redis-rs `parse_protocol` and match its comparison
  (likely `eq_ignore_ascii_case`).

## API polish (non-breaking, land before final)

- A1 `src/lib.rs:826`: add `CacheMetrics` to the prelude. Every `metrics()` caller needs
  the type; today it requires a second import.
- A2 `src/lib.rs:223` vs `src/lib.rs:2039,2050`: the trait doc claims
  `ConcurrentCached::cache_get`/`cache_set` are `#[must_use]` but the attributes are
  absent. Add the attributes (DEC-5).
- A3 `src/stores/redis.rs:684,1141-1148`: add explicit `#[source]` to
  `RedisCacheError::{Redis,Pool}` and `RedisCacheBuildError::Pool` (currently relying on

- A4 `src/stores/redis.rs:700-708`: `RedisCacheBuildError::Resp2DowngradeWithClientSideCaching`
  is only constructible with `redis_async_cache` but is unconditionally public. Gate it
  behind the feature (DEC-4).

## Decisions

- DEC-1: keep `&K` (not `Q: Borrow`) on `ConcurrentCached`/`ConcurrentCachedAsync`
  lookups. Recommendation: keep. Generalizing requires `ShardHasher` and the redis/redb key
  serialization to hash/encode `Q` consistently with `K`, which is a redesign; the sync
  `Cached` trait keeps `Borrow` flexibility. Cost of deferring: stuck until 4.0. Document
  the rationale on the trait (P8).
  - agree, keep
- DEC-2: fix B1 via a non-breaking inherent method on `LruCache` returning the displaced
  `(K, V)`, not by changing the `Cached::cache_set` signature. Recommendation: inherent
  method; the trait change would ripple through every implementor for one internal need.
  - agree, inherent method
- DEC-3: consolidate CHANGELOG rc.1-rc.5 into one `[3.0.0]` section. Recommendation: yes.
  Users migrate 2.x -> 3.0; intra-rc renames (e.g. `CachedAsync` -> `CachedGetOrSetAsync`)
  are noise in a final-release changelog. Keep the rc git tags for archaeology.
  - agree
- DEC-4: gate `Resp2DowngradeWithClientSideCaching` behind `redis_async_cache`.
  Recommendation: gate now. Nominally breaking (a match arm without the feature stops
  compiling) but the variant is unreachable there and the enum is `#[non_exhaustive]`;
  this is the last cheap window.
  - agree, gate
- DEC-5: resolve the `#[must_use]` doc/code mismatch by adding the attributes to
  `ConcurrentCached::cache_get`/`cache_set`. Recommendation: add attributes (warning-only,
  non-breaking, matches the documented contract).
  - agree
- DEC-6: no deprecated `cached::proc_macro` shim module for the removed 2.x import path.
  Recommendation: clean break; the unresolved-import error is searchable and the migration
  guide covers it. (The `time`/`size` attribute rename errors are already excellent.)
  - agree
- DEC-7: cut an rc.6 with the behavior fixes before 3.0.0 final, or go straight to final.
  Recommendation: rc.6. B1/B2 change observable behavior (callback key identity, expiry
  boundary); one short rc keeps the final tag identical to a published artifact.
  - agree rc.6

## Implementation plan

One PR per phase; phases 1-3 can proceed in parallel.

1. Behavior fixes: B1-B5 with regression tests (B1 needs a test where the lookup key and
   stored key are distinguishable; B4/B2 are small lock-scope moves).
2. API polish: A1-A4, T3.
3. Docs: D1-D4 (migration guides + changelog entry), P1-P8, C1-C2.
4. Tests: T1-T2.
5. Release: `/release review` spot-check of the new diff, then `/release 3.0.0-rc.6` (per
   DEC-7), soak, then `/release 3.0.0` with the changelog consolidation (DEC-3) and a final
   `cargo readme` regeneration.

## Verified clean

Cross-type parity: trait method sets are parallel across `Cached`/`ConcurrentCached`/
`ConcurrentCachedAsync` (get/set/remove/remove_entry/delete/clear/reset/reset_metrics/
size); builder vocabulary (`max_size`/`ttl`/`ttl_secs`/`ttl_millis`/`hasher`/`on_evict`/
`shards`/`build`) is uniform; every in-memory `build()` returns `Result<_, BuildError>`
with store-specific errors only for the IO stores.

Consumer experience (external crate, default and minimal feature sets): all root imports
work (`cached::cached`, stores, builders, `BuildError` nameable in signatures); the
`time`/`size` -> `ttl_secs`/`max_size` rename errors and the missing-feature guard errors
(`async`, `time_stores`) are clear and actionable; rustc suggests `CachedExt` when the
trait import is missing; naive constructor guesses (`LruCache::new(100)`,
`TtlCache::new(duration)`, `ShardedUnboundCache::new()`) all compile and run.

Concurrency: lock ordering across shards is consistent (no cross-shard acquisition);
sharded metrics aggregation has no double-counting; `cache_reset_metrics` zeroes inner and
outer counters; the `sync_writes = "by_key"` bucket hasher is per-static randomly seeded.
